would validate but match nothing on the live page.

### Playwright E2E

`e2e/m9-aeo.spec.ts` — 3-test serial suite (gated on
`E2E_ADMIN_EMAIL` + `E2E_ADMIN_PASSWORD`):

1. **Seed + publish a post with faq + 3 steps + tldr.** Login → New
   post → fill title → land on edit. PATCH `content_json` directly
   via the admin API (same approach as M5-core spec — sidesteps the
   editor + inspector path which is already covered by the M3 spec).
   The fixture has 1 heading + 1 paragraph with `role='tldr'` + 3
   paragraphs with `role='step'` + 1 faq. Reload editor; click
   Publish; assert `Status: Published`.
2. **Public `/blog/[slug]` inlines all 5 JSON-LD scripts.** Incognito
   context. Retries up to 5x for ISR propagation. Reads every
   `<script type="application/ld+json">`, asserts `length === 5`,
   parses each, asserts the `@type` set is exactly `{Article,
   BreadcrumbList, FAQPage, HowTo, WebPage}`. Asserts FAQPage has 1
   `mainEntity` with the seeded question, HowTo has 3 steps with
   positions `[1, 2, 3]`, and the Speakable cssSelector is
   `['[data-block-id="t-1"]']`. Spot-checks that
   `[data-block-id="t-1"]` and `[data-block-id="s-1"]` are visible
   on the page.
3. **Cleanup: unpublish + delete.** Same dialog flow as
   `e2e/m4-images.spec.ts` — opens the delete confirmation, types
   the slug, submits, confirms redirect to `/admin/`.

`afterAll` re-logs-in and DELETEs the seeded post via API even on
mid-suite failure, so the dev DB doesn't leak rows between runs.

## 8. Manual-verify path (5-step golden flow)

Assumes the M2 + M3 + M4 + M5 surfaces are merged into `cms-dev` (or
the BE M9-AEO cherry-picks have applied). Postgres on `:5433`, dev
server on `:3000`, admin creds seeded.

1. **Seed an AEO-rich post.** Login at `/admin/login` → New post.
   Title `M9 smoke`. Open the editor, type a paragraph (will become
   the tldr) + a heading + 3 paragraphs (will become steps) + an FAQ
   block (slash menu → faq). Save, publish.
2. **Set roles via the inspector.** Click the first paragraph; in
   the right-rail inspector dropdown set Role to `tldr`. Click the
   first of the 3 step paragraphs; set Role to `step` — note the
   one-line hint "3+ step blocks emit HowTo schema." appearing under
   the dropdown. Repeat for the other two step paragraphs. Save.
3. **Public render — count + types.** Visit `/blog/m9-smoke` in an
   incognito window. View source. You should see FIVE
   `<script type="application/ld+json">` tags directly inside
   `<body>`, in this order: Article → BreadcrumbList → FAQPage →
   HowTo → WebPage (Speakable).
4. **Spot-check the data-block-id anchors.** In DevTools, run
   `document.querySelectorAll('[data-block-id]').length`. The number
   should equal the count of blocks on the page (one anchor per
   block — heading + paragraphs + faq + key-takeaway, etc).
5. **Negative case.** Delete the FAQ block, clear all step roles,
   clear the tldr role; republish. Visit the post again — only TWO
   `<script type="application/ld+json">` tags remain (Article +
   BreadcrumbList). The three M9-AEO scripts are absent because their
   builders returned `null`.

## 9. Known gaps / open questions

1. **Speakable `@id` is unset.** BE flagged this in `m9-aeo-be-2026-05-07.md`
   §7.2: the Speakable `WebPage` shape on the page now coexists with
   the M5-core Article's `mainEntityOfPage: { @type: 'WebPage', @id: <url> }`,
   and some validators may complain about two `WebPage` shapes
   referring to the same canonical URL without disambiguation. FE
   does not have control over this — the fix would be in the BE
   builder (e.g. `@id: <canonical>#speakable`). Defer to TPM whether
   to land in v1.5 or after Tester signal.
2. **HowTo threshold is hard-coded at 3 in BE.** Posts with 2
   well-formed steps + a clear sequence won't get HowTo markup. PRD
   doesn't define a threshold. Same call-out as in BE handoff §7.3.
3. **Inspector hint is text-only.** The hint "3+ step blocks emit
   HowTo schema." surfaces the role-→-schema link without exposing
   how many step blocks the *current* document has. A future
   enhancement could count step blocks in the document state and
   show e.g. "Currently 2 of 3 needed for HowTo." — that requires
   plumbing a count from the editor's serialized state into the
   inspector, which is outside this dispatch's scope.
4. **`data-block-id` is unconditional**, including for code blocks
   that are unlikely to be Speakable targets. The cost is a few
   extra bytes per block in the DOM; the benefit is a uniform
   contract Speakable selectors can rely on. This matches BE §7.1's
   recommendation ("the renderer must emit `data-block-id` on each
   rendered block").
5. **E2E spec not executed in this dispatch.** Same caveat as
   M3/M4/M5. The spec compiles; Tester should run it under their
   seeded admin DB.
6. **`<PostJsonLd>` test-only helper duplicates the page route's
   emit pattern.** If a future change to `app/blog/[slug]/page.tsx`
   alters how the AEO scripts are rendered (e.g. wrapping in a
   `<Suspense>` boundary, or moving to a `<Head>` component), the
   unit tests will pass while production diverges. The E2E spec
   covers the actual route, so the gap is bounded.
